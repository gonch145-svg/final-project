# Predicting Competitive Pokémon Strength Using Machine Learning

## Overview

Competitive Pokémon players tend to select teams based on what's already popular — copying usage trends and avoiding unfamiliar Pokémon. But **usage is not the same as strength**. Strong Pokémon get overlooked simply because nobody has tried them yet, creating a market inefficiency that data can help expose.

This project trains machine learning models on Pokémon base stats to predict competitive viability — and then applies those models to the full Pokémon roster to surface **hidden gems**: Pokémon that should be competitive based on their stats, but are currently underused.

---

## Hypothesis

> Pokémon with higher total stats, strong bulk, and higher speed are more likely to be competitively viable — even when controlling for current usage levels.

**Result: Supported.** Total stats and bulk score are the two strongest predictors of competitive viability by feature importance. Speed contributes meaningfully when combined with other features. The model achieves 91% recall and an AUC-ROC of 0.80, confirming that these stat-based features do predict competitive strength beyond what usage trends alone would suggest.

---

## Datasets

Two datasets are used:

**1. Competitive Dataset** (`pokemon_competitive_analysis.csv`)
- 1,303 Pokémon with recorded competitive usage history
- Smogon VGC and Worlds VGC usage data across 2022, 2023, and 2024
- Base stats (HP, Attack, Defense, Sp. Attack, Sp. Defense, Speed), type, abilities, legendary/mythical status
- Used to train and evaluate the models

**2. Full Pokémon Dataset** (`pokemon_list.csv`)
- 1,072 Pokémon across all generations
- Base stats only — no usage history
- Used to apply the trained model and identify hidden gems

### Target Variable

The target variable `high_usage` is defined as being in the **top 25% by average usage** across Smogon VGC and Worlds VGC tournaments over three years (2022–2024).

- Not Strong (0): 977 Pokémon (75%)
- Strong (1): 326 Pokémon (25%)

Usage is an imperfect proxy for strength — this is exactly the gap the project tries to expose. The model learns what statistically strong Pokémon look like, then finds Pokémon that match that profile but are being ignored.

## Notebooks

### 1. `data_cleaning.ipynb`

Loads both datasets and prepares them for analysis.

**Key steps:**
- Standardises column names and formats across both datasets
- Converts usage percentage strings to numeric values
- Fills missing values (e.g. `type2` → `"none"`, `ability2` → `"none"`)
- Computes average usage across Smogon and Worlds columns for each year
- Creates the binary target variable `high_usage` (top 25% by average total usage)
- Aligns column naming between both datasets (e.g. `sp_attack` → `sp_atk`)
- Exports cleaned files: `pokemon_competitive_clean.csv` and `pokemon_full_clean.csv`

---

### 2. `EDA.ipynb`

Explores the relationship between Pokémon stats and competitive usage.

**Key findings:**
- No single stat strongly predicts competitive usage on its own
- Total stats has the strongest linear correlation with usage (r ≈ 0.21)
- Speed has a weak standalone correlation (r ≈ 0.13), but matters in combination with other features
- Legendary status has the strongest categorical effect (~0.48 point-biserial correlation)
- The top competitively used Pokémon include Incineroar, Flutter Mane, Rillaboom, Amoonguss, and Zacian-Crowned

**Takeaway:** Competitive strength is driven by combinations of features, not individual stats. This motivates the use of machine learning over simple rule-based selection.

---

### 3. `modeling.ipynb`

The main notebook. Covers feature engineering, model training, evaluation, and the hidden gems analysis.

#### Feature Engineering

Five composite features are created on top of the raw base stats, each capturing a competitive archetype:

| Feature | Formula | Competitive meaning |
|---|---|---|
| `offense_score` | Attack + Sp. Attack | Total offensive threat |
| `defense_score` | Defense + Sp. Defense | Balanced bulk |
| `bulk_score` | HP + Defense + Sp. Defense | Raw survivability |
| `speed_tier` | Quantile bin (0–3) of Speed | Which priority bracket a Pokémon belongs to |
| `is_dual_type` | 1 if type2 ≠ "none" | Better coverage and type resistances |

#### Models Trained

Three classifiers are trained on the same features, train/test split, and class weighting:

| Model | Best for | Key trade-off |
|---|---|---|
| Logistic Regression | Discovery — cast a wide net | Highest recall, lower precision |
| Random Forest | Stability — reliable middle ground | Most stable CIs, lowest recall |
| Gradient Boosting | Precision — only flag sure bets | Highest precision, lowest recall |

All models use `class_weight='balanced'` (or equivalent) to handle the 75/25 class imbalance. Hyperparameters are tuned using 5-fold stratified cross-validation with F1 on the Strong class as the scoring metric.

#### Model Results

| Model | Accuracy | Precision | Recall (Strong) | F1 (Strong) | AUC-ROC |
|---|---|---|---|---|---|
| **Logistic Regression** | 0.705 | 0.454 | **0.908** | **0.605** | 0.804 |
| Gradient Boosting | 0.759 | 0.512 | 0.646 | 0.571 | 0.829 |
| Random Forest | 0.770 | 0.538 | 0.538 | 0.538 | 0.829 |

#### Bootstrap Confidence Intervals (95%, n=1,000)

Bootstrap resampling was used to assess model stability:

| Model | Recall CI | CI Width |
|---|---|---|
| Logistic Regression | [0.833 – 0.971] | 0.137 |
| Gradient Boosting | [0.667 – 0.872] | 0.205 |
| Random Forest | [0.517 – 0.754] | 0.238 |

Confidence intervals overlap significantly — no model is statistically dominant. However, Logistic Regression has the highest recall floor and the narrowest interval, making it the most reliable choice for this task.

#### Final Model: Logistic Regression

Selected for its highest recall (91%), most stable confidence intervals, and best alignment with the project objective: finding strong Pokémon, not just avoiding false positives. Missing a genuinely strong Pokémon is a worse error than flagging a weak one.

#### Hidden Gems

The trained model is applied to all 1,072 Pokémon in the full dataset. Pokémon predicted as strong but with low or no usage are classified as hidden gems:

- **Group A — Tracked but underused (194 Pokémon):** Appear in competitive data but were rarely picked. High model confidence suggests their stats warrant more attention. Top examples include Arceus, Blissey, Kyurem, and Goodra.
- **Group B — No usage data (167 Pokémon):** No recorded competitive history. Low usage may reflect format restrictions or simply being overlooked. Top examples include Eternamax Eternatus, Zygarde Complete, and Primal Kyogre.

---

## Key Conclusions

1. **The hypothesis is supported.** Total stats, bulk, and speed — in combination — reliably predict competitive viability.
2. **No single stat is enough.** Competitive strength is multi-dimensional, which is why composite features outperform individual stats.
3. **Logistic Regression outperforms more complex models here.** The relationship between base stats and viability is largely linear, which plays directly to Logistic Regression's strengths. More expressive models can hurt recall on the minority class with a small, imbalanced dataset.
4. **361 hidden gems identified.** 194 are tracked but underused; 167 have no competitive history at all. These represent actionable opportunities for team builders willing to look beyond usage trends.

## Next Steps

The current model is a solid baseline, but several extensions could meaningfully improve its predictive power and practical usefulness.

**Better target variable.** In future iterations, incorporating match-level or tournament win-rate data would allow the model to predict actual competitive success rather than relying on usage as a proxy, enabling a clearer distinction between popularity and true performance.

**Incorporating abilities and movesets.** Base stats capture a Pokémon's raw potential, but abilities (e.g. Intimidate, Drought) and signature moves often determine whether that potential translates to competitive viability. Adding these as features — even encoded categorically — would give the model more of what experienced players actually consider when building teams.

**Team synergy modelling.** The current model treats each Pokémon independently. In practice, competitive viability depends heavily on what teammates a Pokémon is paired with. A next step would be modelling pairs or full team compositions, capturing synergies like speed control, entry hazard support, or type coverage partnerships.

**Tracking meta shifts over time.** The model is trained on 2022–2024 data and treats the meta as static. A time-aware model — trained on rolling windows or with temporal weighting — could detect which Pokémon are rising or falling in viability as the meta evolves, making predictions more actionable for players preparing for upcoming tournaments.

**Format-specific models.** Smogon Singles and VGC Doubles reward very different stats and roles — bulky support Pokémon dominate VGC in a way they rarely do in Singles. Training separate models per format, rather than averaging usage across both, would produce sharper predictions for each competitive context.

**Explainability for individual predictions.** The current model tells you a Pokémon is strong, but not specifically why. Adding SHAP values would allow players to see which stats contributed most to any individual Pokémon's predicted probability — making the model a practical team-building tool, not just a ranking system.

---

## Limitations

- **No movesets or abilities in battle context.** Base stats alone cannot capture everything — a Pokémon's moves, ability, and held item significantly affect competitive viability.
- **No team synergy.** The model treats each Pokémon independently and cannot account for how Pokémon work together on a team.
- **Usage may reflect popularity, not just strength.** The target variable conflates the two, meaning some "strong" Pokémon in the model's eyes may simply be fashionable.
- **Static meta assumption.** The model is trained on 2022–2024 data and cannot adapt to shifts in the competitive meta.
